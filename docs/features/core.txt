# zen
- simple abstraction / minimal implementaion.
- no namespaces in zaal. instead we have modules. (each file is a module)
- functions in zaal is the last layer of abstraction.
- there's no recursion.
- there are only two forms of iteration: for loops & while loops.
- most of the issues should be catch at compile.
- avoid adding features that do the same job but in different ways.
- zaal compiler is stuipd. you need to tell him everything he needs to know.
  - he doesn't "assume" anything for you.
- any mechanism that is outside the control of a compiler should be minimize.
  - there's no inference.
- zaal is not object oriented!
  - there's no operator overloading.
  - there's no polymorphism.
  - there's no inheritance.
    - this encourage users to use composition over inheritance.
- there's no gc.
- there's no macro.
- there's no template.
- there's no alias.
- there's no type punning.
- it's not possible to link with c/c++ code or any other programming langauge.
- runtime should be very small.

- no support for dynamic libraries.
  - all the libraries and binaries should statically build.
- there's no exception.
  - zaal relies on error codes(Result).
    - a Result is just a struct with 2 items:
      struct result
      {
        enum i8 status: { SUCCESS, ERROR }
        string message: "";
      }
    - RESULTs should be check on caller -> otherwise a compiler error will raise.
- identifiers should contain only english letters. (no underscore, no digits)

# keywords
- import, pub
- if, while, for
- addr, ptr, transfer

# comments
There are 2 types of comments:
- code comments: //
- doc comments: ///

# operators
  - equality: "="
  - assignment: ":"
  - less then: "<"
  - less then equal: "<="
  - greater then: ">"
  - greater then equal: ">="
  - array indices: "[]"
  - discard: "_" (to ignore the Result of a function)
  - not: "!"

# primitive types
  - bool(1 byte)
  - char(1 byte) --> it has both char value and integer value of that char.
  - s8, (signed) u8 (unsigned), f8 (float) --> you can't use these as characters, these are just numbers.
  - s16, u16, f16
  - s32, u32, f32
  - s64, u64, f64
  - string
  - ssize, usize

# file extensions and tools
- .zl for zaal files.
- zc is zaal compiler.
- zf is zaal formatter.
- zlsp is zaal lsp.

# memory allocation
allocation on the heap is possible, but it always returns an "owned" poiner or memory, not raw pointer.
  - derefrencing an "owned" memory is possible.
  - derefrencing anything else leads to compiler error. (like variables on stack, etc..)
  - you can't allocate memory without assigning it to any variable. -> compiler error.
    - that object, called: owned memeory variable.
  - when owned memoery variable goes out of scope, memoery belongs to it will free automatically.
    - no manuall memory management needed.
  - ownership can be transfer with transfer keyword.
    ptr test : allocate(10);
    ptr justAPointer : test;
    ptr transferedPointer : transfer test;

# strings
It's just an array of chars with a length.
exmaple:
  string simpleString = "hi". (which is equivalent to: array<char> arrayOfChars = {'h', 'i'})

# enums
  - it defines a set of named constant values.
  - they should have explicit types -> otherwise compiler error.
    - acceptable types: s8, u8, s16, u16, s32, u32, s64, u64
  - enums are not integers! and they don't cast implicitly to integers. -> otherwise compiler error.

# map<KEY, VALUE>
it's a sorted associative container that contains key-value pairs with unique keys.
  example:
    map<string, i32> mapItems{ { "CPU", 10 }, { "GPU", 15 }, { "RAM", 20 } };
    mapItems.erase("CPU"); // this will erase the first pair
    mapItems.append("KEYBOARD", 12); // this will append this pair to end of map

# structs
  - it's an aggregator type that can hold differnt types of data.
  - this allows to pack variables with behaviours(functions) together.
    - they can be public or private. (there's no protected keyword)
  - struct padding SHOULD NOT be hidden and implict. it should be explicit.
    - if you compile for 64 bit systems, struct padding is 8 bytes.
    - if you compile for 32 bit systems, padding is 4 bytes.
    - compiler should stop you from compiling is alignment is not met.
    - you can pad the struct manually by creating the fake bytes, or you can use a function provided by standard library called: pad(NUMBER)
      which accepts a number as argument
        (1, 2, 3) --> For 32 bit system.
        (1, 2, 3, 4, 5, 6, 7) --> for 64 bit systems.
  example:
      {pub} struct <STRUCT_NAME>
      {
        enum i8 status: { SUCCESS, ERROR }
      }

# jungle<SIZE, T, U, V, ...>
It's fixed SIZE of consecutive items of the different types.
  example:
    jungle<3, i32, string, bool> myJungleVar : { 1, "cat", true };
    myJungleVar.remove(); // it removes that last item from the jungle

    jungle<2, string, string> anotherJungleVar : {"day", "night"};
