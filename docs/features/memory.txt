# memory
- each process on operating system has it's own memory address space. (stack + heap)
  - processes can't access the heap(or stack) from other processes. if they want to share data, they should use IPC techniques.
- each process can have different threads.
  - each thread has it's own stack.
  - threads inside a process use the same heap.

- each function has it's own stack frame.
  - inside functions you can use memory from the heap.

- stack: is a LIFO data structure. and it increase downward in memory region. 
  - memory allocated on stack goes out of scope automatically when the function returns.
  - any pointer to local variables are invalid after function return.
  - if you allocate huge amount of data on stack, you'll have stack overflow.

- heap: is huge junk of memory that doesn't have any layout. it increases upward in process memory region.
  - it's the job of allocators to efficiently allocate/deallocate memory from it.
  - heap can be easily fragmented when a lot of allocation happen on it.
  - allocation on heap has cost. (finding the free portiont, fragmentation, etc..) -> it's vary based on time, so it's not deterministic.
  - when you allocated on heap, you're responsible to de-allocate, otherwise you will have memory leak in your process.

# zaal memory management
- we can't only rely on stack for memory.(otherwise we easily get stackoverflow) we "need" heap for large mount of data.
- but we need to be sure that allocation on the heap doesn't introduce all types of heap allocation issues.

zaal memory allocation will be like this:
- all types of data define on stack by default.
- if you want to allocate them on heap, just prepend the declaration with "heap" keyword.
  example:
    int a; // allocated on stack
    heap int b; // allocated on heap
    heap array<int> myArray : {1, 2}; // allocated on heap
    array<int> myArray : {1, 2}; // allocated on stack

- passing data to child functions is "const refrence" by default.
  - if you want to mutate data in child functions, prepnd data as "mut"
    pub fn test() {
      array a<int> : {1, 2};
      myFunc(mut a); // mutable stack refrence

      heap array b<int> : {1, 2};
      myFunc(mut b); // mutable heap refrence

      int c : 1;
      myFunc(a); // const stack refrence
    }
  - there's no copy when you pass variable to another function. (it's always either: "const refrence" or "mut refrence")

- you can pass heap allocated data to parent functions.
  - to do that, you need to transfer the ownership of those data.
  - stack allocated CAN'T be pass to parent functions --> compiler error!

- all variables (no matter on stack or heap), will destroy when the function goes out of scope.
  - if you want to keep the heap memory, when you pass the variable, you need to transfer the ownership.
    fn test() {
      heap array a<int> : {1, 2}
      myFunc(transfer a); // transfer ownership to a child
    }

    fn testTwo(array &a) {
      heap array b<int> : {1, 2}
      a = transfer b; // transfer ownership to parent
    }
  - you CAN'T transfer the ownership of stack data. this concept is just for heap allocated data.
    - because stack data will destroy automatically when function ends.

- ownership of data variables are tied to the function surrunding to them.
  - when the function ends, all heap data will be deallocate(unless you transfer their ownership!)
- derefencing any type of variable is possible. (if it's heap allocated to stack allocated)
  - derefencing can never leads to errors, because:
    - you can't derefnce an out of scope stack variable -> compiler error.
    - you can't derence a heap allocated varible that is no longer exists -> compiler error.
- you can't allocate memory without assigning it to any variable. -> compiler error.
  - that object, called: owned memeory variable.
- when owned memoery variable goes out of scope, memoery belongs to it will free automatically. (unless you transfer the ownership to another function)
  - no manuall memory management needed.
- ownership can be transfer with "transfer" keyword. (ownership tied to functions not variables!)
  heap test : allocate(10); // OK
  heap testTwo : test;      // compiler error! two objects can't point to the same memory address.
  heap testThree : transfer test; // compiler error! you should transfer ownership to either child or parent function.
- heap allocated data, contains these metadata:
  - address: the address on the heap.
  - length: the lenght of the allocated memory on the heap.
  - ownedFunc: the owned function name. (when you transfer a function, this property will change)
  - firstFunc: first functions used this memoery.
  - lastFunc: last functions used this memoery.
